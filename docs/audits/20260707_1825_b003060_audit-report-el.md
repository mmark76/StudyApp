## 7. Κρίσιμα ευρήματα

### 7.1 P1 — Το progress restore μπορεί να αντικαταστήσει settings/content

Το Progress page παρουσιάζει backup ως copy of progress. Στην πράξη, το backup περιλαμβάνει `cardProgress`, `studySessions` και ολόκληρο το `settings` table. Το restore καθαρίζει πλήρως τα τρία tables και μετά εισάγει τα δεδομένα του backup.

Αυτό σημαίνει ότι ένα restore μπορεί να αντικαταστήσει:

- appearance settings,
- imported chapters,
- imported flashcards,
- saved study material links,
- άλλα future settings.

Το πρόβλημα δεν είναι μόνο τεχνικό. Είναι και UX/data-integrity πρόβλημα, γιατί ο χρήστης μπορεί να πιστεύει ότι επαναφέρει μόνο progress, ενώ στην πραγματικότητα επαναφέρει ευρύτερη κατάσταση εφαρμογής.

**Impact:** Υψηλό. Πιθανή απώλεια ή αντικατάσταση user content/settings.  
**Likelihood:** Μεσαίο προς υψηλό, ειδικά αν ο χρήστης κάνει backups σε διαφορετικά στάδια χρήσης.

**Πρόταση διόρθωσης:**

Να χωριστούν καθαρά τα backup modes:

1. `progressBackup` — μόνο card progress και sessions.
2. `contentBackup` — imported units, flashcards και links.
3. `settingsBackup` — appearance/preferences.
4. `fullBackup` — όλα τα παραπάνω, με σαφή προειδοποίηση.

Πριν από restore πρέπει να εμφανίζεται summary:

- πόσες κάρτες progress θα αντικατασταθούν,
- πόσα sessions,
- πόσα imported chapters/flashcards,
- πόσα links,
- αν θα αλλάξουν settings,
- αν το restore είναι merge ή replace.

---

### 7.2 P1 — Το review queue μπορεί να παραλείψει κάρτες

Το Review page υπολογίζει `dueCards` από live Dexie query. Όταν ο χρήστης βαθμολογεί μία κάρτα, η κάρτα ενημερώνεται και παύει να είναι due. Έτσι αλλάζει η live λίστα `dueCards` κατά τη διάρκεια του session.

Παράδειγμα:

1. Αρχική ουρά: `[A, B, C]`.
2. Ο χρήστης απαντά το `A`.
3. Το `A` φεύγει από την due list.
4. Η νέα λίστα γίνεται `[B, C]`.
5. Αν το index αυξηθεί από `0` σε `1`, το app δείχνει `C`.
6. Το `B` παραλείπεται.

**Impact:** Υψηλό για την αξιοπιστία του spaced repetition.  
**Likelihood:** Υψηλό, εφόσον η ουρά αλλάζει live.

**Πρόταση διόρθωσης:**

Στην έναρξη review session να δημιουργείται snapshot:

```ts
const [sessionCardIds] = useState(() => dueCards.map((card) => card.id));
```

ή καλύτερα να υπάρχει explicit `ReviewSession` state που κρατά:

- `startedAt`,
- `cardIds`,
- `currentIndex`,
- `answeredCardIds`,
- `completedAt`.

Το session πρέπει να προχωρά πάνω στο snapshot, όχι στην live due list.

### 7.3 P2 — Quiz double-submit / duplicate session risk

Στο quiz answer flow δεν υπάρχει ισχυρό lock τύπου `useRef`. Αν ο χρήστης πατήσει πολύ γρήγορα δύο επιλογές ή γίνει διπλό tap σε mobile, υπάρχει κίνδυνος:

- λάθος score,
- διπλή καταχώρηση session,
- race condition πριν ενημερωθεί το React state.

**Impact:** Μεσαίο.  
**Likelihood:** Μεσαίο, ιδιαίτερα σε mobile.

**Πρόταση διόρθωσης:**

Να προστεθεί lock:

```ts
const answering = useRef(false);

async function answer(option: string) {
  if (!question || finished || answering.current) return;
  answering.current = true;
  try {
    // existing answer logic
  } finally {
    answering.current = false;
  }
}
```

Εναλλακτικά, disabled option buttons αμέσως μετά το πρώτο click/tap.

### 7.4 P2 — Backup validation χρειάζεται αυστηρότερο schema

Το backup validation ελέγχει βασική μορφή, αλλά όχι αρκετά βαθιά.

Χρειάζονται επιπλέον έλεγχοι:

- duplicate `cardId`,
- duplicate setting keys,
- valid ISO timestamps,
- non-negative repetitions/lapses,
- valid interval ranges,
- supported setting keys,
- maximum backup size,
- maximum array lengths,
- relationship checks όπου είναι πρακτικό.

**Πρόταση:** Δημιουργία `backupSchema.ts` με explicit runtime validation και test cases για malformed input.

### 7.5 P2 — CSV import χωρίς header validation

Ο CSV parser διαβάζει απλώς την πρώτη γραμμή ως header και μετά κάνει positional parsing. Το UI ζητά από τον χρήστη να μη μεταβάλει column headings, όμως ο κώδικας δεν φαίνεται να επαληθεύει ότι τα headings είναι σωστά.

**Impact:** Μεσαίο.  
**Risk:** Ο χρήστης μπορεί να αλλάξει στήλες ή να εξάγει CSV από άλλη εφαρμογή με διαφορετική σειρά στηλών, οδηγώντας σε λανθασμένο import.

**Πρόταση:** Exact header validation με καθαρό error message:

```text
Expected columns: Chapter number, Question, Answer, Tags
```

Για chapters:

```text
Expected columns: Number, Title, Objectives, Summary, Key Terms
```

---

### 7.6 P2 — Deterministic spreadsheet IDs μπορεί να αλλοιώσουν progress

Τα imported flashcard IDs δημιουργούνται από `unitNumber` και row index. Αν ο χρήστης αλλάξει σειρά rows ή εισάγει νέα row στη μέση, οι κάρτες μπορεί να αλλάξουν ID ή να κληρονομήσουν λάθος progress.

**Πρόταση:**

- Προαιρετική στήλη `id` στα CSV templates.
- Ή stable generated ID από normalized content:

```text
card-${unitNumber}-${hash(question + answer)}
```

- Ή import preview που δείχνει creates/updates/conflicts.

### 7.7 P3 — File validation κυρίως με extension/MIME

Το local file import ελέγχει supported extensions και MIME types. Αυτό είναι λογικό για αρχική έκδοση, αλλά όχι πλήρης content validation.

**Πρόταση:**

- Για PDF: έλεγχος magic bytes `%PDF` ή δοκιμή με `PDFDocument.load` όπου έχει νόημα.
- Για images: optional `createImageBitmap` validation.
- Για text/CSV: max line count, encoding handling και clearer parse errors.

### 7.8 P3 — PWA update behaviour πιθανόν υπερβολικά επιθετικό

Το service worker register γίνεται με immediate behaviour και auto update. Για εφαρμογή με local study sessions, καλό είναι να μη γίνεται update που μπορεί να επηρεάσει ενεργή χρήση χωρίς ορατή ενημέρωση.

**Πρόταση:**

Να εμφανίζεται update banner:

```text
A new version is available. Update now / later.
```

Ο χρήστης πρέπει να έχει έλεγχο, ειδικά όταν βρίσκεται σε review ή quiz session.

### 7.9 P4 — Build ID δεν είναι reproducible

Το build identifier περιλαμβάνει ημερομηνία και ώρα build. Αυτό είναι χρήσιμο για debug, αλλά μειώνει reproducibility.

**Πρόταση:**

```text
version + commit short SHA
```

και το build time να μένει optional ή μόνο σε CI metadata.

## 11. Security και privacy

Δεν εντοπίστηκε προφανές remote data leakage στο επίπεδο του ελεγχθέντος κώδικα. Η εφαρμογή φαίνεται να αποθηκεύει τοπικά progress, files, links και settings.

- Δεν υπάρχει account system.
- Δεν υπάρχει backend dependency.
- Τα local files αποθηκεύονται στον browser.
- Τα cloud links αποθηκεύουν μόνο title και URL.
- Τα URLs περιορίζονται σε HTTP/HTTPS.
- Το local file open χρησιμοποιεί object URLs και προστασία `noopener noreferrer`.

Κύριο privacy/data risk:

Η απώλεια δεδομένων δεν είναι κυρίως από διαρροή προς server, αλλά από:

- clearing browser/site data,
- private browsing,
- browser storage eviction,
- restore που αντικαθιστά δεδομένα,
- ανεπαρκώς διακριτά backup modes.

Η εφαρμογή ήδη προειδοποιεί τον χρήστη για αρκετά από αυτά. Χρειάζεται όμως πιο ασφαλές backup/restore UX.
